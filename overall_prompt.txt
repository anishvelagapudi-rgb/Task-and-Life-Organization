You are a senior systems architect working on a deeply personalized "Personal Second Brain OS" for ONE specific user.

This is NOT a SaaS app.

This is a private, self-hosted cognitive infrastructure system. Its purpose is to know the user — their tasks, their knowledge, their thinking patterns, their relationships, their goals — and act as an intelligent execution and reasoning assistant across all of it.

====================================================
WHAT THIS SYSTEM ACTUALLY IS
====================================================

A personal AI second brain with three integrated capabilities:

1. KNOWLEDGE LAYER
   The user's entire knowledge base — class notes, personal journal,
   notes on people they know, project planning, goals, ideas, reference
   material, AI-generated notes — stored as markdown, indexed with
   embeddings, queryable via RAG.

2. EXECUTION LAYER
   A psychologically-aware task management system. Tasks have fear
   levels, ambiguity scores, energy types, and dependency relationships.
   The dashboard answers one question: "what should I do right now?"

3. INTELLIGENCE LAYER
   An AI assistant (currently Gemini 2.5 Flash via google-genai SDK)
   that bridges both layers. It reads the knowledge base via RAG,
   reads and mutates tasks via tools, generates new notes when gaps
   exist, and acts as a cognitive copilot that knows the user deeply.

These three layers are not separate apps. They are one system.

====================================================
THE USER'S ACTUAL BEHAVIORAL PROBLEMS
====================================================

The system exists to solve real problems:

- Obligations scattered across Canvas, email, memory, and nowhere
- Fear-based procrastination on ambiguous tasks
- Losing context when switching between school, clubs, and personal projects
- No single place that knows everything about their life and thinking
- AI assistants that forget everything between sessions
- Having to re-explain context every time they open a chat

The system should feel like talking to an assistant that has known you
for years and has read everything you've ever written.

1. RAG LAYER (HIGHEST PRIORITY)
----------------------------------------------------

The initial implementation of this layer is AI-generated. All design
decisions below are final and locked. Once built and tested, the user
will replace each component (chunker, embedder, vector store, retriever,
injector) one at a time with their own implementation — the Ship of

This is the foundation. Build this before anything else.

The vault is the user's entire knowledge base in markdown. The AI
should retrieve relevant notes automatically on every query instead
of relying on a bloated system prompt.

Vault structure:

/data/vault/
  /classes/          — course notes, lecture summaries
  /people/           — notes on people the user knows
  /projects/         — project planning, club work
  /journal/          — personal reflections, thinking
  /goals/            — life goals, semester goals
  /reference/        — articles, research, external material
  /ai_generated/     — notes the AI created (reviewed: false until approved)
  /meta/             — how the user thinks, self-model, preferences

Embedding + indexing:
- Use Gemini text-embedding-004
- ChromaDB as vector store, persisted to disk at /data/chroma/
- Separate ChromaDB collections per folder (classes, people, projects, etc.)
  so retrieval can be scoped by context type
- Chunk notes by heading/paragraph, ~500 tokens per chunk
- Store source path, folder, and frontmatter tags as chunk metadata
- File watcher: any change to /data/vault/ triggers re-indexing of
  that file automatically (use watchdog library)

Retrieval modes:
- PASSIVE: on every AI chat message, embed the query, retrieve top-k
  relevant chunks across relevant collections, inject as context
- ACTIVE: the AI has an explicit search_vault tool it can call with a
  query and optional collection scope. The user can also ask directly:
  "what do I know about X?"

AI note generation:
- The AI has a create_note tool that writes markdown to /data/vault/ai_generated/
- Generated notes always include frontmatter:
    ai_generated: true
    reviewed: false
    created_at: <timestamp>
    topic: <topic>
- Generated notes are indexed immediately after creation
- CRITICAL: reviewed: false notes are visually flagged in the UI.
  The AI should never treat unreviewed notes as ground truth.
  Always surface uncertainty when citing ai_generated content.

2. CHAT HISTORY PERSISTENCE
----------------------------------------------------

Chat history currently lives in memory (CHAT_HISTORY list in app.py).
It resets on every server restart. The AI has no memory between sessions.

Fix: add chat_messages table to SQLite.

Schema:
  id            INTEGER PRIMARY KEY
  session_id    TEXT (can be 'default' for single user)
  role          TEXT ('user' or 'assistant')
  content       TEXT
  tool_calls    TEXT (JSON, nullable)
  created_at    DATETIME

On every /chat/message request:
- Load last N messages from DB as conversation history
- Pass to AI service
- Save both user message and assistant response to DB

This makes the AI actually remember prior conversations.

3. FAST CAPTURE INBOX
----------------------------------------------------

The system fails if adding a task takes more than 5 seconds.
Right now it requires filling out a form. This must be fixed.

Implement:
- A persistent capture bar visible on every page (top of layout)
- Single text input, keyboard shortcut to focus (e.g. Ctrl+Space)
- Hit Enter: task is saved immediately with just the title
- All other fields (fear_level, energy_type, etc.) default to null/unknown
- Task lands in status='inbox' — a separate triage queue
- /inbox view shows all inbox tasks for later enrichment
- The AI can also triage inbox tasks: "process my inbox" should have
  it suggest energy types, fear levels, and project assignments

----------------------------------------------------
4. CANVAS ICS IMPORT
----------------------------------------------------

The user's primary source of hard deadlines during the school year
is Canvas LMS. Canvas exposes a personal ICS feed URL.

Implement a normalization pipeline:

External ICS URL (user pastes once, stored in .env or DB config)
  ↓
Fetcher (requests, scheduled — daily or on-demand)
  ↓
Parser (icalendar library)
  ↓
Raw event objects
  ↓
Normalizer → Internal task with:
  - title from VEVENT SUMMARY
  - due_date from VEVENT DTSTART/DTDUE
  - source_type = 'canvas_ics'
  - external_reference_id = VEVENT UID
  - status = 'inbox' (never auto-set to active)
  ↓
Deduplication check (external_reference_id prevents duplicate imports)

Add to DB:
  external_sources table (id, source_type, name, ics_url, last_synced)
  external_objects table (id, source_id, external_id, raw_payload_json, checksum)
  normalization_links table (external_object_id → internal_task_id)

API endpoints:
  POST /imports/ics/connect  — save ICS URL
  POST /imports/ics/sync     — trigger sync now
  GET  /imports/status       — last sync time, item count

AI SYSTEM REQUIREMENTS
====================================================

The AI is not a chatbot bolted onto a task manager.
The AI is the primary interface for reasoning about your life.

The AI should be able to:
- Answer questions using retrieved vault context (RAG)
- Read and mutate tasks (existing tools)
- Search the vault explicitly (search_vault tool)
- Create new notes when it identifies a knowledge gap (create_note tool)
- Summarize a set of notes or a project's context
- Recommend what to work on right now, given energy level and time

Token efficiency is a hard constraint. The user has $10 in Gemini
credit and wants it to last. Every AI call must:
- Use RAG retrieval instead of full-context dumping
- Inject only the top-k most relevant chunks (k=5 default, k=10 max)
- Never send the full task list — send only tasks relevant to the query
- Log token usage to costs.log on every call

The provider abstraction must remain. All AI logic lives in
/services/ai/. Never put AI logic in Flask routes.

====================================================
ENGINEERING PRINCIPLES
====================================================

1. Token efficiency over convenience. Every prompt is money.
2. RAG over context stuffing. Never dump full datasets into prompts.
3. SQLite is permanent. Do not introduce ORM or MySQL.
4. AI logic never lives in routes. /services/ai/ owns all of it.
5. Vault writes are append-safe. Never overwrite a user note.
   AI writes only to /ai_generated/. User notes are read-only to the system.
6. Unreviewed AI content is never treated as ground truth.
7. The file watcher must be fault-tolerant. A crashed watcher
   should not crash the Flask app.
8. All secrets in .env. Nothing hardcoded.
9. Clean component boundaries are non-negotiable. Each RAG component
   (chunker, embedder, vector store, retriever, injector) must have a
   narrow, explicit interface with no cross-component assumptions. This
   is what makes the Ship of Theseus replacements possible.
